物料分配需求

超卖问题

  • 问题描述：节点A下面有两个有流量统计的子节点B和C。当B节点流量用完后，如果此时并发导致B又进来一个决策。当前的做法是，数据更新失败，没有抛异常，返回结果成功；则导致当前节点超卖。由于当前对流量统计，并没有要求很精确，并且两种决策结果都一样，所以，暂时没有导致问题。
  • 解决方法：决策中的超卖问题，涉及了多个节点，正常情况下，应该是走到C节点，而不是抛异常报错。解决方案：
    • 上个节点A感知超卖，然后，再走C节点。
    • B抛超卖异常，外层捕获后，重新决策。

指标梳理

操作员维度

  • AB过滤掉：否、空白：

      • 对应期的用户数：有3、6、9、12、18、24分期数。

全局维度

  • AB过滤掉：否、空白；并去重后客编：
    • 成功户数：COUNT(E)，即客编的数量。
    • 分期总期次(汇总的平均数)：AVG(AC)，即分期期数的平均数

    • 消费总和：SUM(Q)，即消费金额求和
    • 分期金额总和：SUM(AD)，即分期金额求和
    • 处理户数：distinct COUNT(E)，去重后求总数

分期金额AD阈值

  • AB过滤掉 否、空白； 按照客编维度对AD求和的值 在6个区间里面
    • 户数：COUNT(E)，即6个区间的客户数.

注：分期金额阈值[0,5000] [5000,10000] [10000,20000] [20000,50000] [50000,100000] [100000, 正无穷]；左闭右开。

决策配置管控二期：dd配置管控和慧脑融合

进行了细分评审，并进入开发阶段。

细分阶段的主要问题点：

  • 对类目和业务单元关系：最初的设计要把业务单元弱化掉；因子、动作、规则在二级类目下管理，在全局范围使用；决策树直接在三级类目管理。经过激烈讨论后，最终方案是运行时以业务单元为主，配置态以类目为主。这样，后台代码可以兼容，同时配置人员只需要感知业务单元。
  • 拷贝模式和引用模式：最初的设计要把因子、动作、规则全局使用，并且因子是引用模式，这样不可避免的导致全局更改的不可控。最终的设计是，运行时跟一期保持一致；配置态以二级类目为准。

开发进度：

  • 配置管控相关部分迁移到新的bundle：mng-facade
  • facade接口设计和定义完成80%，决策流接口设计和定义还未完成。
  • 建表/DO层/Repository:完成70%。

下周对主流程进行开发，同时，也要兼容以前代码。

欢迎使用 MWeb

首先介绍一下 MWeb 是什么，MWeb 是专业的 Markdown 写作、记笔记、静态博客生成软件。MWeb 使用的是 Github Flavored Markdown 语法，在使用 MWeb 前，建议你一定要去 MWeb 官网首页 看一下介绍视频，MWeb 官网也做了比较详细的帮助，建议你也看一下大概内容，帮助的网址为：http://zh.mweb.im/help.html。

需要您重点注意的

MWeb 是有两种模式的：外部模式和文档库模式。MWeb 为了满足所有 Markdown 使用需求，设计了两种模式！外部模式用于打开和编辑所有本地 Markdown 文档。另外为了方便用 Markdown 记笔记，MWeb 设计了文档库模式，文档库中的文档也支持一键生成静态博客，以便于分享。关于两个模式的情况，还是建议您去看一下官网的视频和帮助。

帮助我们改进 MWeb

如果你喜欢 MWeb，想让它变得更好，你可以：

  1. 推荐 MWeb，让更多的人知道。
  2. 给我们发反馈和建议：coderforart+233@gmail.com
  3. 在 Mac App Store 上评价 （如果是在 MAS 上购买的话）。

网格智能运营

网格0817

本周主要是联调和发布阶段。负责重点区域区域和复合人群的开发联调。

  • LBS相关的负责结构树组装，对外屏蔽；同时，二叉树切为多叉树模式。
    • LBS数据集查询：在da进行包装。之前，开发的时候忽略了热力图模式，周四进行了兼容支持。
    • 创建重点区域或LBS人群：最初开发的时候使用了二叉结构树模式，本周发现：当使用100个POI的时候，树的深度是99，组装出来的sql将会嵌套99层，导致性能问题。修复为使用DS的多叉树能力，并在DA将树封装为对象。
  • nomo的一些需求整理：对nomo的一些功能进行了整理，下周同步给一风，再统一给nomo提需求。其中有个问题，nomo不支持不规则区域和POI条件的复用，商量后，在DA层进行了网格拆分。最后，在语法验证的时候，发现nomo将两类地理位置信息做了and关系，不符合产品需求。跟pd商量后，在页面将两类分开建立。
  • da优化整理：本期功能完成的同时，还有一些地方需要进一步优化改进，对其做了整理
  • 数据一致性：均挚本次提供了pgsql的runner，同时在两个数据集上面进行了验证。但是，由于pgsql精度更高一些，将会导致这两个数据集跟其他数据，口径对不上。因此，本次仍然全部采用ads。后续，考虑将一个独立模块进行引流。

其他

  • mif查询报错：周五上午，mif调用ds由十几个报错，和若道一起排查后，发现是参数中心配置没有获取到。找崇幻确认后，发现参数中心在mrchdt和其自身系统，均无报错，并且数据也加载到了。虽然报错数量不多，但是该机器为灰度机器，调用量总量很少，从该机器的比例来看，并不低。重启该机器后，无报错出现；后面，找pe确认，该机器也并没有问题。该问题，还需要后续的继续观察。
